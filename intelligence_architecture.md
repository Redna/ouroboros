# Ouroboros Intelligence Architecture

This document outlines the core mechanisms used by Ouroboros to manage cognitive load, maintain context continuity, and ensure stable interaction with Large Language Models (LLMs).

## 1. Message Normalization
To prevent API errors (specifically `400 Bad Request` errors common in certain LLM providers when receiving consecutive system messages), Ouroboros implements a normalization layer in `llm_interface.py`.

- **Mechanism**: When multiple `system` role messages appear sequentially, they are merged into a single `system` message with newline delimiters.
- **Purpose**: Ensures compatibility across diverse model backends (e.g., LlamaCpp, OpenAI-compatible APIs).

## 2. Cognitive Shedding (Context Management)
As the conversation progresses, the token count increases. To prevent "cognitive collapse" (context window overflow), Ouroboros utilizes a `shed_heavy_payloads` mechanism. This is a selective pruning process rather than a simple truncation.

### A. Temporal Retention
- **Full Context**: The last $N$ messages (`RETAIN_FULL_LAST_N`) are kept in their entirety to ensure the agent maintains immediate high-definition awareness of its recent actions and state.
- **Historical Pruning**: Messages older than $N$ are processed for "weight reduction."

### B. Content-Specific Shedding Strategies

#### I. Reasoning/Thinking Shedding
- **Target**: `thinking` or `reasoning_content` fields in assistant messages.
- **Logic**: Once a reasoning turn is no longer part of the immediate context, the heavy reasoning blocks are stripped. This preserves the *result* of the thought without the *process*, saving massive amounts of tokens.
- **Constraint**: Thinking is never stripped from the *absolute last* assistant message to avoid breaking prefill compatibility.

#### II. Telemetry (HUD) Archiving
- **Target**: `<ouroboros_hud>` XML blocks.
- **Logic**: Instead of keeping raw telemetry, Ouroboros replaces the block with a high-signal summary: `[SYSTEM LOG: Historical Telemetry Archived: [HEARTBEAT METRICS]]`.
- **Purpose**: Maintains the "knowledge" that telemetry occurred without the token cost of repeating the HUD.

#### III. Interrupt Pruning
- **Target**: `<system_interrupt>` XML blocks.
- **Logic**: Replaces detailed interrupt content with a minimal indicator: `[SYSTEM LOG: Historical [CREATOR/SYSTEM] Interrupt Handled]`.
- **Purpose**: Maintains the timeline of events (continuity) while shedding the redundant payload of the interrupt itself.

#### IV. Tool Output Truncation
- **Target**: Large strings in `tool` role messages.
- **Logic**: If a tool output exceeds `TOOL_OUTPUT_TRIM_CHARS`, it is truncated to a small preview followed by a length indicator.
- **Purpose**: Prevents massive data dumps (e.g., from `ls -R` or large file reads) from consuming the entire context window.

#### V. Tool Call Argument Trimming
- **Target**: Large arguments within `tool_calls` (e.g., large code snippets in a `patch_file` call).
- **Logic**: Arguments exceeding `TOOL_ARG_TRIM_CHARS` are replaced with an archival string: `(... X characters of [key] archived ...)`.
